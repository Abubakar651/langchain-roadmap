┌──────────────────────────────────────────────────────────────────────────────┐
│ 1.1  WHAT IS A VECTOR DATABASE?                                              │
└──────────────────────────────────────────────────────────────────────────────┘

A vector database is a specialised database that stores data as high-dimensional
numerical vectors (called embeddings) and supports fast similarity search over them.

Unlike a traditional database:
  Traditional DB  →  stores rows, columns, integers, strings
                     searches by EXACT match:  WHERE name = 'Alice'

  Vector DB       →  stores float arrays like [0.12, -0.87, 0.44, 0.03, ...]
                     searches by SIMILARITY:   "find things closest to this vector"

Real-world analogy:
  Imagine a library. A traditional database is like a card catalogue — you find
  a book only if you spell the title exactly right. A vector database is like a
  wise librarian — you describe what you're looking for ("a story about courage
  during war") and she finds books that MEAN the same thing, even if none of the
  words match exactly.


┌──────────────────────────────────────────────────────────────────────────────┐
│ 1.2  WHY DO WE NEED VECTOR DATABASES?                                        │
└──────────────────────────────────────────────────────────────────────────────┘

Problem 1 — LLMs have a limited context window
  GPT-4        : ~128,000 tokens  ≈ ~100 pages
  LLaMA 3.3   : ~128,000 tokens  ≈ ~100 pages
  Your document: could be 1,000+ pages (a legal contract, a research corpus)

  You CANNOT paste the entire document into the prompt. You need a way to
  find and retrieve only the relevant parts first.

Problem 2 — LLMs can hallucinate
  When an LLM doesn't know the answer, it makes one up. If you give it
  actual source material as context, it stays grounded in facts.

Problem 3 — LLMs' training data is outdated
  A model trained in 2024 knows nothing about 2025 events. Vector databases
  let you feed it current, private, or specialised knowledge.

Problem 4 — Keyword search is not smart enough
  SQL LIKE '%machine learning%' misses documents that say "neural networks",
  "deep learning", or "AI models" — all semantically equivalent.

  Vector search finds all of them because they embed to similar vectors.

Solution → RAG + Vector Database:
  Store your knowledge as vectors → retrieve the relevant bits → give to LLM
  This is the most practical and widely-used LLM architecture today.


┌──────────────────────────────────────────────────────────────────────────────┐
│ 1.3  HOW DOES A VECTOR DATABASE WORK?                                        │
└──────────────────────────────────────────────────────────────────────────────┘

Step 1 — Embed
  Every piece of text is passed through an embedding model which outputs a
  vector. Example:

    "The cat sat on the mat"  →  [0.12, -0.33, 0.87, 0.04, -0.21, ...]
                                  (384 numbers for MiniLM, 1536 for OpenAI)

Step 2 — Index
  The vectors are stored in a special data structure called an index.
  The index is optimised for one task: "given a new vector, find the k
  closest vectors already in the index."

  Common index types:
    Flat (exact)  — checks every vector, 100% accurate, slow at scale
    IVF           — divides vectors into clusters, searches relevant clusters
    HNSW          — Hierarchical Navigable Small Worlds, graph-based, very fast

Step 3 — Query
  At query time:
    a) Embed the user's question into a vector
    b) Run ANN (Approximate Nearest Neighbour) search in the index
    c) Return the top-k most similar vectors (and their original text)

Step 4 — Use
  The retrieved texts become the "context" passed to the LLM.

┌──────────────────────────────────────────────────────────────────────────────┐
│ 1.5  FAISS DEEP DIVE                                                         │
└──────────────────────────────────────────────────────────────────────────────┘

FAISS = Facebook AI Similarity Search (by Meta AI Research)

Key characteristics:
  - Written in C++ with Python bindings (very fast)
  - Supports both CPU and GPU
  - Can handle billions of vectors
  - Multiple index types for speed/accuracy trade-off

LangChain FAISS API:

  # Create
  db = FAISS.from_documents(docs, embeddings)
  db = FAISS.from_texts(["text1", "text2"], embeddings)

  # Search
  results = db.similarity_search(query, k=4)
  results = db.similarity_search_with_score(query, k=4)  # includes distance

  # Add more
  db.add_documents(new_docs)

  # Persist
  db.save_local("my_index/")
  db = FAISS.load_local("my_index/", embeddings, allow_dangerous_deserialization=True)

  # Use in a chain
  retriever = db.as_retriever(search_type="mmr", search_kwargs={"k": 4})

FAISS distance metrics:
  L2 (Euclidean) — default. Lower = more similar. Range: 0 to ∞
  Cosine         — use normalize_embeddings=True in HuggingFaceEmbeddings first
  Inner Product  — used with normalized vectors (equivalent to cosine)

MMR (Maximal Marginal Relevance):
  Problem: Top-4 results might all say the same thing (redundant)
  MMR fixes this by balancing:
    - Relevance to the query
    - Diversity (avoid near-duplicate chunks)
  Use: search_type="mmr" in as_retriever()

┌──────────────────────────────────────────────────────────────────────────────┐
│ 2.1  WHAT ARE EMBEDDINGS?                                                    │
└──────────────────────────────────────────────────────────────────────────────┘

An embedding is a list of floating-point numbers that represents the *meaning*
of a piece of text in a mathematical space.

Example (simplified to 3D for illustration):
  "king"   →  [0.9,  0.1,  0.8]
  "queen"  →  [0.85, 0.9,  0.8]
  "man"    →  [0.9,  0.1,  0.2]
  "woman"  →  [0.85, 0.9,  0.2]
  "apple"  →  [0.1,  0.3,  0.05]

Notice: king and man are close. queen and woman are close. Apple is far from all.

In reality:
  - all-MiniLM-L6-v2  → 384 dimensions
  - text-embedding-3-small (OpenAI) → 1536 dimensions
  - Every dimension captures some latent aspect of meaning


┌──────────────────────────────────────────────────────────────────────────────┐
│ 2.2  WHY ARE EMBEDDINGS IMPORTANT?                                           │
└──────────────────────────────────────────────────────────────────────────────┘

They let computers understand language *semantically*, not literally.

Without embeddings (keyword search):
  Query: "fix car engine"
  Miss:  "repair automobile motor"     ← same meaning, different words

With embeddings (semantic search):
  Query: "fix car engine"
  Finds: "repair automobile motor"     ← similar vector, found correctly
  Finds: "troubleshoot vehicle powertrain" ← also semantically close

This is revolutionary for search, recommendations, clustering, and RAG.


┌──────────────────────────────────────────────────────────────────────────────┐
│ 2.3  HOW ARE EMBEDDINGS CREATED?                                             │
└──────────────────────────────────────────────────────────────────────────────┘

Embedding models are neural networks (usually transformers) trained on massive
text datasets with a specific objective: make similar texts produce similar vectors.

Training objectives include:
  - Contrastive learning: pull similar sentence pairs together, push dissimilar apart
  - Masked language modelling: predict missing words
  - Natural language inference: understand entailment and contradiction

The model processes text through multiple transformer layers. The final layer's
output (a [CLS] token or mean-pooled representation) becomes the embedding vector.

In code (LangChain):
  from langchain_huggingface import HuggingFaceEmbeddings

  model = HuggingFaceEmbeddings(
      model_name="all-MiniLM-L6-v2",
      model_kwargs={"device": "cpu"},
      encode_kwargs={"normalize_embeddings": True},
  )

  # Single query
  vector = model.embed_query("What is machine learning?")
  # → list of 384 floats

  # Batch of documents (more efficient)
  vectors = model.embed_documents(["doc1 text", "doc2 text", "doc3 text"])
  # → list of 3 × 384 floats

┌──────────────────────────────────────────────────────────────────────────────┐
│ 2.5  COSINE SIMILARITY VS L2 DISTANCE                                        │
└──────────────────────────────────────────────────────────────────────────────┘

Two ways to measure how close two vectors are:

COSINE SIMILARITY:
  - Measures the ANGLE between two vectors (ignores magnitude/length)
  - Formula: cos(θ) = (A · B) / (|A| × |B|)
  - Range: -1 (opposite) to 1 (identical)
  - 1.0 = perfectly similar, 0.0 = unrelated, -1.0 = opposite
  - Best when vectors are not normalised to unit length

L2 (EUCLIDEAN) DISTANCE:
  - Measures the straight-line DISTANCE between two points in space
  - Formula: sqrt(Σ(a_i - b_i)²)
  - Range: 0 (identical) to ∞
  - 0 = identical, larger = more different
  - FAISS uses L2 by default

Practical note:
  If you set normalize_embeddings=True in HuggingFaceEmbeddings,
  all vectors have length = 1, so cosine similarity and L2 distance
  give equivalent rankings. This is recommended.

Example:
  sentence A: "I love pizza"
  sentence B: "Pizza is my favourite food"
  sentence C: "The stock market fell 3% today"

  A vs B: cosine ~ 0.87  (very similar)
  A vs C: cosine ~ 0.12  (unrelated)

┌──────────────────────────────────────────────────────────────────────────────┐
│ 3.1  WHAT IS RAG?                                                            │
└──────────────────────────────────────────────────────────────────────────────┘

RAG = Retrieval-Augmented Generation

It is an architecture that enhances LLM responses by retrieving relevant
information from an external knowledge base before generating an answer.

Introduced by: Facebook AI Research, paper "Retrieval-Augmented Generation
               for Knowledge-Intensive NLP Tasks" (2020)

Simple definition:
  RAG = Smart Search + LLM

  Instead of asking the LLM "what do you know about topic X?",
  you first FIND the relevant documents yourself, then ask the LLM
  "given THIS specific text, answer this question."


┌──────────────────────────────────────────────────────────────────────────────┐
│ 3.2  WHY IS RAG IMPORTANT?                                                   │
└──────────────────────────────────────────────────────────────────────────────┘

Problem 1 — HALLUCINATION
  LLMs sometimes generate confident-sounding but completely false answers.
  RAG grounds the LLM in real source text, dramatically reducing this.

Problem 2 — KNOWLEDGE CUTOFF
  LLMs are trained on data up to a certain date and know nothing after.
  RAG lets you add current information without retraining the model.

Problem 3 — PRIVATE / PROPRIETARY DATA
  Your company's internal documents, contracts, codebase, medical records —
  the LLM was never trained on them. RAG bridges this gap.

Problem 4 — COST OF FINE-TUNING
  Retraining or fine-tuning a large model costs thousands of dollars and
  requires deep ML expertise. RAG is cheap and fast to set up.

Problem 5 — AUDITABILITY
  With RAG you can show the user exactly which document chunks were used
  to generate the answer. This is critical in healthcare, legal, finance.

  LLM alone:  "Answer: X" (no way to verify or trace)
  RAG:        "Answer: X [Source: contract.pdf, page 7]" (fully traceable)


┌──────────────────────────────────────────────────────────────────────────────┐
│ 3.3  HOW DOES RAG WORK? (STEP-BY-STEP)                                      │
└──────────────────────────────────────────────────────────────────────────────┘

There are two phases: INDEXING (done once) and QUERYING (done per question).

════ INDEXING PHASE ════════════════════════════════════════════════════════════

Step 1 — Load Documents
  Read your source material: PDFs, Word docs, web pages, databases, etc.
  LangChain document loaders handle this:
    PyPDFLoader      → PDFs
    WebBaseLoader    → websites
    CSVLoader        → CSV files
    DirectoryLoader  → entire folders

Step 2 — Split into Chunks
  Documents are split into smaller overlapping pieces.
  WHY? Two reasons:
    a) Embedding quality degrades for very long texts
    b) You want to retrieve the SPECIFIC section that answers the question,
       not the entire 200-page document

  CHUNK SIZE trade-off:
    Too small (100 chars) → loses context, fragments ideas
    Too large (5000 chars) → retrieves too much, dilutes relevance
    Sweet spot: 500–1500 characters with 10–15% overlap

  OVERLAP:
    Last 150 chars of chunk N = first 150 chars of chunk N+1
    This prevents ideas from being cut off at chunk boundaries

Step 3 — Embed Chunks
  Each chunk → embedding model → dense vector
  All vectors stored in memory (FAISS) or sent to a cloud DB (Pinecone)

════ QUERYING PHASE ════════════════════════════════════════════════════════════

Step 4 — Embed the User's Question
  User asks: "What is the refund policy?"
  → embedding model → [0.23, -0.11, 0.67, ...]  (same 384-dim space)

Step 5 — Retrieve Relevant Chunks
  FAISS performs ANN search: find top-k vectors closest to the query vector
  Returns: the 4 most semantically relevant text chunks from the documents

Step 6 — Build the Prompt
  The retrieved chunks are inserted into a prompt template:

    "Answer using only the context below.
     Context: [chunk1] [chunk2] [chunk3] [chunk4]
     Question: What is the refund policy?
     Answer:"

Step 7 — Generate
  The full prompt is sent to the LLM (Groq → LLaMA 3.3 70B)
  The LLM reads the context and generates a grounded answer.

Step 8 — Return to User
  Answer is displayed with source references so the user can verify.


┌──────────────────────────────────────────────────────────────────────────────┐
│ 3.4  RAG vs FINE-TUNING                                                      │
└──────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────┬──────────────────────────┬──────────────────────────┐
│ Aspect              │ RAG                       │ Fine-tuning              │
├─────────────────────┼──────────────────────────┼──────────────────────────┤
│ Cost                │ Very low                  │ $$$–$$$$                 │
│ Time to deploy      │ Hours                     │ Days to weeks            │
│ Knowledge updates   │ Real-time (just re-index) │ Requires retraining      │
│ Hallucination       │ Low (grounded in source)  │ Can still hallucinate    │
│ Source attribution  │ Yes (shows exact chunks)  │ No                       │
│ Private data        │ Yes                       │ Yes (but risky)          │
│ Reasoning quality   │ Good                      │ Can be better for style  │
│ Best for            │ Knowledge Q&A, documents  │ Style, tone, new tasks   │
└─────────────────────┴──────────────────────────┴──────────────────────────┘

USE FINE-TUNING when you want the LLM to behave differently (new tone, format,
                   or specialised task not well represented in training data).

USE RAG when you want the LLM to know specific facts from your documents.
           This is the right choice for 80%+ of real-world use cases.


┌──────────────────────────────────────────────────────────────────────────────┐
│ 3.5  TEXT SPLITTING — WHY AND HOW                                            │
└──────────────────────────────────────────────────────────────────────────────┘

WHY SPLIT AT ALL?
  1. Embedding models have a max input length (512 tokens for MiniLM)
  2. Retrieving a 100-page doc is useless; retrieving 3 relevant paragraphs works
  3. Smaller chunks = more precise retrieval

RecursiveCharacterTextSplitter — HOW IT WORKS:
  It tries to split on a hierarchy of separators, in order:
    ["\n\n", "\n", ". ", " ", ""]

  First tries to split on double newlines (paragraph breaks).
  If that makes chunks too big, splits on single newlines.
  If still too big, splits on sentences. Then words. Then characters.
  This respects natural text structure as much as possible.

Parameters:
  chunk_size    = 1000   # max characters per chunk
  chunk_overlap = 150    # characters shared between adjacent chunks

Example:
  ┌─────────────────────────────────────────────────────────┐
  │ Original text (3000 chars):                             │
  │ "Python is a language... [paragraph 1, 800 chars]       │
  │  LangChain is a framework... [paragraph 2, 900 chars]   │
  │  FAISS is a library... [paragraph 3, 1300 chars]"       │
  └─────────────────────────────────────────────────────────┘
                          ↓ split (chunk_size=1000, overlap=150)
  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
  │ Chunk 1      │ │ Chunk 2      │ │ Chunk 3      │ │ Chunk 4      │
  │ Para 1 full  │ │ last 150 of  │ │ Para 2 full  │ │ last 150 of  │
  │ (800 chars)  │ │ para1 +      │ │ (900 chars)  │ │ para2 +      │
  │              │ │ start para2  │ │              │ │ start para3  │
  └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘

Note the OVERLAP: each chunk shares 150 characters with the previous.
This ensures no key sentence falls entirely at the boundary of two chunks.


┌──────────────────────────────────────────────────────────────────────────────┐
│ 3.6  THE FULL RAG PIPELINE WITH CODE                                         │
└──────────────────────────────────────────────────────────────────────────────┘

from langchain_huggingface import HuggingFaceEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_groq import ChatGroq
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# ── 1. Your knowledge ────────────────────────────────────────────────────────
text = """
LangChain is an open-source framework for building LLM applications.
It was created by Harrison Chase in 2022.
Key components: chains, agents, memory, document loaders, vector stores.
"""

# ── 2. Split ─────────────────────────────────────────────────────────────────
splitter = RecursiveCharacterTextSplitter(chunk_size=300, chunk_overlap=50)
chunks = splitter.create_documents([text])

# ── 3. Embed + Index ─────────────────────────────────────────────────────────
embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
db = FAISS.from_documents(chunks, embeddings)

# ── 4. Retriever ─────────────────────────────────────────────────────────────
retriever = db.as_retriever(search_kwargs={"k": 3})

# ── 5. Prompt ────────────────────────────────────────────────────────────────
prompt = ChatPromptTemplate.from_template("""
Answer using only the context below.
Context: {context}
Question: {question}
Answer:""")

# ── 6. LLM ───────────────────────────────────────────────────────────────────
llm = ChatGroq(model="llama-3.3-70b-versatile", temperature=0)

# ── 7. Chain (LCEL) ──────────────────────────────────────────────────────────
def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# ── 8. Ask ───────────────────────────────────────────────────────────────────
answer = chain.invoke("Who created LangChain?")
print(answer)  # → "LangChain was created by Harrison Chase in 2022."

┌──────────────────────────────────────────────────────────────────────────────┐
│ 4.1  PROJECT OVERVIEW                                                        │
└──────────────────────────────────────────────────────────────────────────────┘

WHAT IT DOES:
  A web app where you upload any PDF file(s) and ask questions about them
  in plain English. The app finds the relevant parts and answers using
  the LLaMA 3.3 70B model via Groq's ultra-fast inference API.

FEATURES:
  - Upload multiple PDFs at once
  - Automatic text extraction and chunking
  - FAISS vector index persisted to disk (survives app restarts)
  - Extend the index: upload more PDFs without losing previous ones
  - MMR retrieval for diverse, non-redundant context
  - Source attribution: see exactly which page/file the answer came from
  - Chat history displayed in the UI
  - Adjustable chunk size, overlap, and k (chunks to retrieve)

┌──────────────────────────────────────────────────────────────────────────────┐
│ 4.3  FILE-BY-FILE EXPLANATION                                                │
└──────────────────────────────────────────────────────────────────────────────┘

── pdf_processor.py ────────────────────────────────────────────────────────────

  WHAT: Handles everything to do with reading PDF files and preparing them
        for embedding.

  WHY separate file: The PDF loading/splitting logic is reusable and testable
                     on its own, separate from the vector store or UI.

  KEY FUNCTIONS:

  load_pdf(file_path):
    - Uses LangChain's PyPDFLoader
    - Returns one Document per PDF page
    - Adds filename to each page's metadata
    - WHY PyPDFLoader? It's simple, reliable, and preserves page numbers.
      Page numbers become part of the metadata so you can say "see page 7".

  split_documents(documents, chunk_size=1000, chunk_overlap=150):
    - Feeds pages into RecursiveCharacterTextSplitter
    - Returns smaller chunks with original metadata preserved
    - Adds a chunk_index field to metadata
    - WHY 1000/150 defaults? A good balance — large enough to capture full
      ideas, small enough to be precise when retrieved.

  process_pdfs(file_paths, ...):
    - Orchestrates load + split for multiple files
    - Returns one flat list of all chunks from all PDFs
    - Prints progress to console


── vector_store.py ─────────────────────────────────────────────────────────────

  WHAT: A module that owns the FAISS index — create, load, update, clear.

  WHY separate file: The vector DB logic is independent of the RAG chain
                     and UI. This separation makes testing easier and lets
                     you swap FAISS for Pinecone later with minimal changes.

  KEY DESIGN DECISIONS:

  INDEX_DIR = project/faiss_index/
    - The index is saved here automatically
    - Survives app restarts; you don't re-index every time you reload the app

  EMBEDDING_MODEL = "all-MiniLM-L6-v2"
    - Defined once here; used consistently everywhere
    - normalize_embeddings=True: vectors are unit-length → cosine-equivalent search

  get_embeddings():
    - Always uses the same model config
    - IMPORTANT: the same model that created the index MUST be used to
      query it. Different models produce incompatible vector spaces.

  build_vector_store(documents):
    - Creates a brand-new FAISS index from scratch
    - Overwrites any existing index
    - Use this when uploading the first batch of PDFs

  add_documents(db, documents):
    - Adds new vectors to an existing index without losing old ones
    - Use this when the user uploads additional PDFs later

  get_retriever(db, k=4):
    - Returns a retriever with MMR search
    - MMR = Maximal Marginal Relevance
    - fetch_k=k*3: internally fetches 12 candidates, then picks 4 diverse ones
    - WHY MMR? Avoids returning 4 near-identical chunks about the same sentence


── rag_chain.py ────────────────────────────────────────────────────────────────

  WHAT: Builds the LangChain LCEL pipeline that turns a question into an answer.

  WHY separate file: The chain logic (prompt, LLM, parsing) is distinct from
                     the vector store. This makes it easy to swap the LLM or
                     change the prompt without touching the storage layer.

  KEY COMPONENTS:

  RAG_PROMPT:
    - A ChatPromptTemplate with two variables: {context} and {question}
    - Instructs the LLM to ONLY use provided context (reduces hallucination)
    - Asks for source attribution in the answer
    - Sets fallback: "I couldn't find relevant information..."

  format_docs_with_sources(docs):
    - Formats retrieved chunks as:
        [Source 1: contract.pdf, page 7]
        ... chunk text ...
        ---
        [Source 2: policy.pdf, page 2]
        ... chunk text ...
    - WHY: The LLM can read the source labels and include them in its answer,
      making the response fully attributable.

  build_rag_chain(db, k=4):
    - Creates the full LCEL chain
    - Returns BOTH the chain AND the retriever
    - WHY return retriever separately? So the Streamlit app can invoke it
      independently to get the raw source documents to display to the user.

  ask(chain, retriever, question):
    - Calls chain.invoke(question) → answer string
    - Calls retriever.invoke(question) → list of source Documents
    - Returns both so the UI can display answer + sources


── app.py ───────────────────────────────────────────────────────────────────────

  WHAT: The Streamlit web application — the user-facing layer.

  KEY STREAMLIT CONCEPTS USED:

  st.session_state:
    - Persists data across reruns (Streamlit re-executes the whole script
      on every user interaction)
    - Stores: db, chain, retriever, chat_history, indexed_files
    - Without this, the FAISS index would disappear on every button click

  @st.cache_resource:
    - Caches the FAISS index load so it only runs once per server session
    - Without this, the disk index would be reloaded on every page interaction

  st.file_uploader:
    - Returns UploadedFile objects (in-memory, not on disk)
    - WHY save to tempfile? PyPDFLoader needs a real file path on disk.
      We write the uploaded content to a temporary file, process it, then delete.

  Sidebar:
    - Document upload and management controls on the left
    - Main chat interface on the right
    - This mirrors chat app conventions (like ChatGPT's sidebar)

  Chat history:
    - Stored as list of {"role": "user"/"assistant", "content": "..."}
    - Rendered with st.chat_message() for the familiar chat bubble UI
    - Source chunks stored alongside assistant messages for the expander


┌──────────────────────────────────────────────────────────────────────────────┐
│ 4.4  HOW EACH COMPONENT WORKS TOGETHER                                       │
└──────────────────────────────────────────────────────────────────────────────┘

USER UPLOADS A PDF — SEQUENCE:

  1. app.py: user clicks "Index PDFs" button
  2. app.py: saves uploaded file bytes to a temp file on disk
  3. app.py: calls process_pdfs([temp_path]) from pdf_processor.py
  4. pdf_processor.py: PyPDFLoader reads the temp file → pages
  5. pdf_processor.py: RecursiveCharacterTextSplitter → chunks
  6. app.py: calls build_vector_store(chunks) or add_documents(db, chunks)
  7. vector_store.py: HuggingFaceEmbeddings embeds all chunks
  8. vector_store.py: FAISS.from_documents() creates the index
  9. vector_store.py: db.save_local() writes to faiss_index/ on disk
  10. app.py: calls build_rag_chain(db) → stores chain + retriever in session
  11. app.py: deletes the temp file, shows success message


USER ASKS A QUESTION — SEQUENCE:

  1. app.py: user types question in chat input
  2. app.py: calls ask(chain, retriever, question) from rag_chain.py
  3. rag_chain.py: chain.invoke(question) triggers the LCEL pipeline:
       a) RunnableParallel splits into two parallel branches:
          - "context" branch: retriever.invoke(question)
            → FAISS encodes question → ANN search → top 4 chunks
            → format_docs_with_sources() → string with [Source N: ...]
          - "question" branch: RunnablePassthrough() → same question string
       b) Both outputs fill the ChatPromptTemplate slots
       c) Formatted prompt sent to ChatGroq (Groq API → LLaMA 3.3 70B)
       d) LLM response passed through StrOutputParser → plain string
  4. rag_chain.py: retriever.invoke(question) called again → source Documents
  5. app.py: displays answer in chat bubble
  6. app.py: displays source chunks in an expander below the answer


┌──────────────────────────────────────────────────────────────────────────────┐
│ 4.5  LANGCHAIN LCEL (PIPE SYNTAX) EXPLAINED                                  │
└──────────────────────────────────────────────────────────────────────────────┘

LCEL = LangChain Expression Language

It uses Python's pipe operator | to chain components together.
Each component is a "Runnable" — it has an invoke() method.

Basic chain:
  chain = prompt | llm | output_parser

  This means:
    1. Take input → feed to prompt → get formatted prompt
    2. Feed formatted prompt → feed to llm → get AIMessage
    3. Feed AIMessage → feed to output_parser → get string

RAG chain (more complex):

  chain = (
      {"context": retriever | format_docs, "question": RunnablePassthrough()}
      | prompt
      | llm
      | StrOutputParser()
  )

  Breaking this down:

  {"context": retriever | format_docs, "question": RunnablePassthrough()}
    → This is a RunnableParallel (dict with Runnables as values)
    → When chain.invoke("my question") is called:
        - "context" branch: retriever gets "my question" → finds docs →
          format_docs() joins them into one string
        - "question" branch: RunnablePassthrough() just forwards "my question"
    → Output: {"context": "chunk1\n\nchunk2...", "question": "my question"}

  | prompt
    → ChatPromptTemplate receives the dict, fills in {context} and {question}
    → Output: ChatPromptValue (a formatted prompt ready for the LLM)

  | llm
    → ChatGroq receives the formatted prompt
    → Sends to Groq API → LLaMA 3.3 70B processes it
    → Output: AIMessage(content="The answer is...")

  | StrOutputParser()
    → Extracts just the text from AIMessage
    → Output: "The answer is..."  (plain Python string)

WHY LCEL?
  - Readable: you can see the whole pipeline at a glance
  - Composable: easy to add/remove/swap components
  - Async support: works with async/await for streaming
  - Debugging: LangSmith can trace every step automatically

│ 5.2  COMMON ERRORS AND FIXES                                                 │
└──────────────────────────────────────────────────────────────────────────────┘

ERROR: ModuleNotFoundError: No module named 'langchain.text_splitter'
FIX:   Use: from langchain_text_splitters import RecursiveCharacterTextSplitter
WHY:   In LangChain v1+, text splitters were moved to a separate package.

ERROR: ModuleNotFoundError: No module named 'torchvision'
FIX:   pip install torchvision
WHY:   Some sentence-transformer models need torchvision even without images.

ERROR: allow_dangerous_deserialization must be True
FIX:   db = FAISS.load_local(path, embeddings, allow_dangerous_deserialization=True)
WHY:   LangChain requires explicit opt-in to load pickled files (security measure).

ERROR: ValueError: Expected EmbeddingModel to produce vectors of size X, got Y
FIX:   Use the SAME embedding model to build and query the index.
WHY:   Different models produce different vector sizes; they're incompatible.

ERROR: Groq API rate limit
FIX:   Add time.sleep(1) between requests, or use a smaller model like gemma2-9b-it
WHY:   Free Groq tier has requests-per-minute limits.

ERROR: PDF text extraction is empty / garbled
FIX:   Some PDFs are image-scans. Use pypdf + pytesseract (OCR) instead.
       pip install pytesseract  (also install Tesseract system package)


┌──────────────────────────────────────────────────────────────────────────────┐
│ 5.3  GLOSSARY                                                                │
└──────────────────────────────────────────────────────────────────────────────┘

ANN (Approximate Nearest Neighbour)
  An algorithm that finds vectors close to a query vector quickly by
  trading a small amount of accuracy for much faster speed.

Chunk
  A small piece of text created by splitting a larger document.
  Typically 500–1500 characters with some overlap with adjacent chunks.

Context Window
  The maximum amount of text an LLM can process in one call.
  LLaMA 3.3 70B: ~128,000 tokens. Older GPT-3: 4,096 tokens.

Cosine Similarity
  A measure of the angle between two vectors. 1.0 = identical direction
  (same meaning), 0.0 = perpendicular (unrelated), -1.0 = opposite.

Dense Vector
  A vector where most/all values are non-zero. Embeddings are dense.
  Contrast with sparse vectors (like TF-IDF) where most values are 0.

Embedding
  A dense numerical representation of text (or images, audio) that
  captures semantic meaning. Produced by an embedding model.

FAISS
  Facebook AI Similarity Search. A C++ library with Python bindings
  for fast similarity search over large sets of vectors.

Fine-tuning
  Continuing the training of a pre-trained model on a specific dataset
  to adapt its behaviour. Expensive but powerful for style/task changes.

Groq
  An AI hardware and API company. Their LPU (Language Processing Unit)
  delivers extremely fast LLM inference (750+ tokens/second).

Hallucination
  When an LLM generates text that is confidently stated but factually
  incorrect. RAG reduces hallucination by grounding answers in source text.

HNSW (Hierarchical Navigable Small Worlds)
  A graph-based ANN algorithm. One of the fastest for high-dimensional
  vector search. Used internally by FAISS and Qdrant.

LCEL (LangChain Expression Language)
  A declarative syntax using the | operator to chain LangChain components
  (Runnables) into pipelines. Supports sync, async, and streaming.

LLM (Large Language Model)
  A neural network trained on massive text data that can generate,
  understand, and transform text. Examples: GPT-4, LLaMA, Claude.

MMR (Maximal Marginal Relevance)
  A retrieval strategy that balances relevance to the query with
  diversity among retrieved results, avoiding redundant chunks.

PyPDFLoader
  A LangChain document loader that uses the pypdf library to extract
  text from PDF files, one Document per page.

RAG (Retrieval-Augmented Generation)
  An architecture that retrieves relevant documents from a knowledge base
  and provides them as context to an LLM before generating an answer.

Retriever
  A LangChain abstraction that takes a query string and returns a list
  of relevant Document objects. Wraps any vector store search method.

Runnable
  Any LangChain component that implements invoke(), batch(), and stream().
  Chains, prompts, LLMs, output parsers, retrievers are all Runnables.

Semantic Search
  Finding documents that are *meaningfully similar* to a query, not just
  textually identical. Powered by vector embeddings and similarity search.

Sentence Transformer
  A class of models trained to produce semantically meaningful sentence
  embeddings. The sentence-transformers library by UKP Lab on HuggingFace.

Session State (Streamlit)
  A dictionary (st.session_state) that persists across reruns of the
  Streamlit script. Used to store the FAISS index and chat history.

Token
  The basic unit of text that LLMs process. Roughly 0.75 words.
  "Hello world!" = 3 tokens. Context windows are measured in tokens.

Vector Space
  A mathematical space where each dimension represents some abstract
  feature of meaning. Similar texts occupy nearby regions of this space.
